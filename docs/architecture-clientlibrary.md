# Architecture: ClientLibrary .NET Network Layer

## Executive Summary

ClientLibrary is a .NET 10 library published as a Native AOT DLL that provides the network communication layer between the MuMain C++ game client and OpenMU game servers. It handles TCP connection management, packet serialization/deserialization using `MUnique.OpenMU.Network.Packets`, and bidirectional function calls across the C++/.NET boundary via exported delegates.

## Technology Stack

| Category | Technology | Version | Notes |
|----------|-----------|---------|-------|
| Runtime | .NET | 10.0 | Native AOT compilation |
| Language | C# | Latest | With unsafe/interop features |
| Packet Library | MUnique.OpenMU.Network.Packets | 0.9.8 | NuGet: XML packet definitions |
| IO Pipeline | Pipelines.Sockets.Unofficial | 2.2.8 | High-performance socket I/O |
| Code Generation | XSLT | 1.0 | 4 transforms: XML → C++/C# |
| Publishing | Native AOT | — | Single DLL, no runtime dependency |

## Architecture Pattern

**Adapter/Bridge Pattern** — Translates between C++ game client and .NET network protocol implementation.

```
┌─────────────────────────┐         ┌─────────────────────────┐
│  MuMain (C++)           │         │  OpenMU Server          │
│                         │         │  (TCP, encrypted)       │
│  Connection.h/cpp       │         │                         │
│  ├── LoadLibrary()      │         │                         │
│  ├── GetFunctionPtr()   │         │                         │
│  └── CallDotNet()  ─────┼────►    │                         │
│                         │    ┌────┼─────────────────────────┤
│  PacketFunctions_*.cpp  │◄───┤    │                         │
│  (callback delegates)   │    │    │                         │
└─────────────────────────┘    │    └─────────────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │  ClientLibrary      │
                    │  (.NET Native AOT)  │
                    │                     │
                    │  ConnectionManager  │
                    │  ├── Connect()      │
                    │  ├── SendPacket()   │
                    │  ├── OnReceive()    │
                    │  └── Disconnect()   │
                    │                     │
                    │  *Functions.cs      │
                    │  (generated C# ext) │
                    └─────────────────────┘
```

## Core Components

### ConnectionManager.cs
- **Role:** Central connection lifecycle manager
- **Responsibilities:** TCP socket management, encryption setup, packet routing
- **Pattern:** Singleton with exported `[UnmanagedCallersOnly]` methods for C++ consumption

### ConnectionWrapper.cs
- **Role:** Wraps the raw connection with convenience methods
- **Responsibilities:** Connection state tracking, reconnection logic, error handling

### Generated Extension Files (*Functions.cs)
- **Generated by:** `GenerateExtensionsDotNet.xslt`
- **Source:** `MUnique.OpenMU.Network.Packets` XML definitions
- **Output files:**
  - `ConnectionManager.ClientToServerFunctions.cs` (257 KB — largest)
  - `ConnectionManager.ChatServerFunctions.cs`
  - `ConnectionManager.ConnectServerFunctions.cs`

### Custom Handler Files
- `ConnectionManager.ClientToServer.Custom.cs` — Manual overrides for game server packets
- `ConnectionManager.ChatServer.Custom.cs` — Manual overrides for chat packets

## Code Generation Pipeline

4 XSLT transforms convert XML packet definitions to both C++ and C# bindings:

| Transform | Output | Target |
|-----------|--------|--------|
| `GenerateExtensionsDotNet.xslt` | `*Functions.cs` | C# extension methods |
| `GenerateFunctionsHeader.xslt` | `PacketFunctions_*.h` | C++ handler declarations |
| `GenerateFunctions.xslt` | `PacketFunctions_*.cpp` | C++ handler stubs |
| `GenerateBindingsHeader.xslt` | `PacketBindings_*.h` | C++ packet struct definitions |

Shared utilities in `Common.xslt`. Triggered by Visual Studio PreBuild event (skipped in CI via `'$(ci)'!='true'`).

## Data Marshaling

| Type | C++ Side | .NET Side | Notes |
|------|----------|-----------|-------|
| Strings | `wchar_t*` (2-byte) | `char*` UTF-16LE | Windows-specific, needs `char16_t` for cross-platform |
| Packets | `byte*` + length | `ReadOnlySpan<byte>` | Raw byte buffers |
| Callbacks | Function pointers | `[UnmanagedCallersOnly]` delegates | Registered at DLL load time |
| Numerics | Native sizes | Matching sizes | Direct pass-through |

## Protocol Coverage

3 server protocols with packet definitions:

| Protocol | XML Source | Packet Count | Use Case |
|----------|-----------|-------------|----------|
| ClientToServer | `ClientToServerPackets.xml` | ~200+ | Main game server communication |
| ChatServer | `ChatServerPackets.xml` | ~20 | Chat, whisper, guild chat |
| ConnectServer | `ConnectServerPackets.xml` | ~10 | Server list, initial connection |

## Build & Deployment

```bash
# Published as Native AOT DLL (no .NET runtime required)
dotnet publish -c Release -r win-x64

# Output: MUnique.Client.Library.dll
# Copied to game executable directory by CMake post-build step
```

The CMake build system auto-detects `dotnet.exe` (Windows) or `dotnet` (Linux) and handles WSL path translation via `wslpath`.

## Cross-Platform Considerations

- **Current:** Windows-only (`wchar_t` = 2 bytes, `LoadLibrary`)
- **Planned:** Linux/macOS support requires:
  - `char16_t` at interop boundary (Linux `wchar_t` = 4 bytes)
  - `dlopen` / `dlsym` for DLL loading
  - `.so` / `.dylib` output instead of `.dll`
  - Runtime identifiers: `linux-x64`, `osx-arm64`, `osx-x64`
