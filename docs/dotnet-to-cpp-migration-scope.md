# .NET to C++ Network Layer Migration Scope

> Assessment date: 2026-04-11. Based on current ClientLibrary architecture (MUnique.OpenMU.Network.Packets v0.9.8, .NET 10 Native AOT).

## Executive Summary

The .NET ClientLibrary is a **serialization + encryption bridge**, not business logic. 93% of its C# code is XSLT-generated boilerplate. The actual hand-written logic is ~525 lines covering TCP transport, encryption setup, and credential handling. Migrating to pure C++ would take **15-22 working days**, eliminate the .NET SDK build dependency, and unblock Android/iPadOS builds (see `mobile-port-feasibility.md`).

**Critical risk:** SimpleModulus encryption must be ported byte-for-byte faithfully — any drift causes server-side packet rejection.

---

## Current Architecture

```
OpenMU Server (TCP/IP, Encrypted)
         ↕
    ClientLibrary (.NET 10, Native AOT)
         ↕ function pointers ([UnmanagedCallersOnly])
    PacketFunctions_*.cpp (C++ wrappers)
         ↕
    MuMain Game Client (C++20)
```

The library has four responsibilities:

| Layer | What It Does | Implemented In |
|-------|-------------|----------------|
| TCP Transport | `TcpClient` -> `SocketConnection` -> `System.IO.Pipelines` | `ConnectionManager.cs`, `ConnectionWrapper.cs` |
| Encryption Pipeline | `PipelinedSimpleModulusEncryptor/Decryptor` + `PipelinedXor32Encryptor` | NuGet: `MUnique.OpenMU.Network` |
| Packet Serialization | 213 functions that fill packet structs and write to pipe | Generated by XSLT from XML packet definitions |
| Packet Framing | Length-prefixed frames, `SetPacketSize()` | `MUnique.OpenMU.Network` |

## Encryption Algorithms to Port

### SimpleModulus (Transport Layer) — HIGH RISK

- Stream cipher operating on fixed-size blocks with modular arithmetic
- Applied to **all bytes** before transmission on encrypted connections (game server)
- Uses pre-shared keys: `PipelinedSimpleModulusEncryptor.DefaultClientKey` / `PipelinedSimpleModulusDecryptor.DefaultClientKey`
- Reference implementation: [OpenMU source](https://github.com/MUnique/OpenMU) — `PipelinedSimpleModulusEncryptor.cs`, `PipelinedSimpleModulusDecryptor.cs`
- **Must match server expectations byte-for-byte** — any deviation causes packet rejection
- Estimated: ~400 lines of C++ (encrypt + decrypt)

### Xor3 (Field-Level Encryption) — LOW RISK

- 3-byte rotating XOR key: `FC CF AB`
- Configurable start offset: 0 (login credentials), 5 (chat messages), 6 (chat tokens)
- Applied to sensitive fields only (username, password, chat tokens, chat messages)
- Estimated: ~30 lines of C++

### Xor32 (Transport Wrapper) — LOW RISK

- Simple 32-bit XOR applied per-byte to packet stream
- Wraps SimpleModulus as outer encryption layer
- Estimated: ~20 lines of C++

### Encryption Pipeline Assembly

```
// Current (.NET):
encryptor = PipelinedXor32Encryptor(
    PipelinedSimpleModulusEncryptor(socketOutput, DefaultClientKey).Writer
)
decryptor = PipelinedSimpleModulusDecryptor(socketInput, DefaultClientKey)

// Target (C++):
// Encrypt path:  raw bytes -> SimpleModulus -> Xor32 -> socket
// Decrypt path:  socket -> SimpleModulus -> raw bytes
// (Xor32 is only on the encrypt side per OpenMU protocol)
```

Connect server connections use `isEncrypted = false` — no encryption pipeline, plain TCP.

## Migration Strategy

### Approach: XSLT Retarget (Recommended)

Modify the existing XSLT templates to generate **direct C++ serialization** instead of .NET interop wrappers. The XML packet definitions remain the single source of truth.

| Approach | Effort | Maintenance | Risk |
|----------|--------|-------------|------|
| **A. Retarget XSLT -> C++ serialization** | 3-5 days | Packet changes auto-propagate via XML -> XSLT -> C++ | Medium — template complexity |
| B. Hand-write 213 C++ functions | 2-3 weeks | Must manually update for any protocol change | Low initially, high ongoing |

Approach A preserves the existing code generation workflow. When the `MUnique.OpenMU.Network.Packets` NuGet updates with new packets, the XSLT regenerates the C++ code automatically.

### What Gets Created

```
NEW:
├── src/source/Network/MuConnection.h/cpp       — TCP socket + receive thread (~300 lines)
├── src/source/Network/SimpleModulus.h/cpp       — Encrypt/decrypt (~400 lines)
├── src/source/Network/Xor3.h                    — Field-level XOR (~30 lines)
├── src/source/Network/Xor32.h                   — Transport XOR (~20 lines)
├── src/source/Network/PacketWriter.h            — Buffer builder for serialization (~100 lines)
├── ClientLibrary/GenerateFunctionsCpp.xslt      — New XSLT: direct C++ serialization (~150 lines)
└── Modified: PacketFunctions_*.h/cpp            — Regenerated (direct write, no .NET calls)
```

### What Gets Deleted

```
REMOVED:
├── ClientLibrary/ConnectionManager.cs           — .NET connection manager
├── ClientLibrary/ConnectionWrapper.cs           — .NET async wrapper
├── ClientLibrary/*.Custom.cs                    — Hand-written .NET send functions
├── ClientLibrary/*Functions.cs                  — Generated C# serialization (7,304 lines)
├── ClientLibrary/MUnique.Client.Library.csproj  — .NET project file
├── ClientLibrary/GenerateExtensionsDotNet.xslt  — C# code generator
├── src/source/Dotnet/PacketBindings_*.h         — Function pointer bindings (678 lines)
├── src/source/Dotnet/Connection.cpp             — .NET interop bridge (510 lines)
├── CMakeLists.txt: dotnet publish step           — ~80 lines of AOT build logic
└── PlatformLibrary.h: dlopen/LoadLibrary shims  — No longer needed
```

### C++ API Preserved

The public `Connection` class API remains identical — callers throughout the game client see no change:

```cpp
// Before and after — same interface
class Connection {
public:
    Connection(const char16_t* host, int32_t port, bool isEncrypted,
               void (*packetHandler)(int32_t, const BYTE*, int32_t));
    ~Connection();
    bool IsConnected();
    void Send(const BYTE* data, int32_t length);
    void Close();
    PacketFunctions_ClientToServer* ToGameServer() const;
    PacketFunctions_ChatServer* ToChatServer() const;
    PacketFunctions_ConnectServer* ToConnectServer() const;
};
```

The `PacketFunctions_*` classes also keep their existing method signatures. Only the implementation changes from "forward to .NET" to "serialize directly and write to socket."

## Collateral Benefits

Removing the .NET dependency eliminates several layers of complexity:

| Removed | Impact |
|---------|--------|
| SIOF workarounds | `ResolvePacketBindings()` with 213 `ReResolve()` calls, `extern` variable dance in `Connection.h` |
| .NET SDK build requirement | No more `dotnet publish --runtime` step; faster CI; simpler developer setup |
| Platform library loading | No `dlopen`/`LoadLibrary`/`GetSymbol` for the managed DLL |
| `MU_DOTNET_LIB_DIR` / `MU_DOTNET_LIB_EXT` CMake plumbing | Build configuration simplified |
| `coreclr_delegates.h` dependency | No CoreCLR hosting headers needed |
| `char16_t` marshaling (`MU_C16`) | Can use UTF-8 internally end-to-end |
| Mobile platform blocker | Unblocks Android (NDK) and iPadOS (Xcode) builds |

### Verification Strategy

SimpleModulus correctness must be verified before integration:

1. **Unit tests with known vectors:** Port the OpenMU C# unit tests for SimpleModulus; compare byte-for-byte output against the .NET reference for the same input
2. **Capture/replay:** Record encrypted packets from the current .NET implementation; verify the C++ port produces identical output for the same input
3. **Live server test:** Connect to OpenMU server, complete login -> character select -> world entry flow

## NuGet Dependency Analysis

The `MUnique.OpenMU.Network.Packets` NuGet (v0.9.8) provides:

| What | Used For | Post-Migration |
|------|----------|---------------|
| XML packet definitions | XSLT code generation source | **Still needed** — input to retargeted XSLT |
| `PipelinedSimpleModulusEncryptor/Decryptor` | Transport encryption | **Replaced** by C++ `SimpleModulus.h/cpp` |
| `PipelinedXor32Encryptor` | Transport XOR layer | **Replaced** by C++ `Xor32.h` |
| `Xor3Encryptor` | Field-level credential encryption | **Replaced** by C++ `Xor3.h` |
| `*Ref` packet structs (`PingRef`, `LoginLongPasswordRef`, etc.) | Span-based packet builders | **Replaced** by XSLT-generated C++ serialization |
| `Pipelines.Sockets.Unofficial` | Socket-to-pipe adapter | **Replaced** by C++ socket I/O |

Post-migration, the XML files from the NuGet package are the only dependency. These can be vendored into the repository to remove the NuGet requirement entirely.
